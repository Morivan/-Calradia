### Расхождения между README и кодом

#### Критические

1. **README указывает Windows-путь `E:\Project`** — проект кроссплатформенный, документация вводит разработчика в заблуждение. Нужно переписать на относительные пути.

2. **VK-интеграция отсутствует на уровне сервисов** — модель `Client` имеет `Source.VK`, в интерфейсе отображаются VK-клиенты из демо-данных, но нет ни `vk.py`-сервиса, ни webhook/polling для VK. Реальные сообщения из VK не поступают.

3. **Кнопки CRM не подключены к API**:
   - «Создать заказ из чата» (`App.tsx:1624`) — кликабельна, но не вызывает `OrderCreateView`.
   - «Сменить статус заказа» (`App.tsx:1625`) — нет handler-а и нет соответствующего endpoint на backend.

4. **Маркетинговый редактор доступен только для чтения** (`App.tsx:1368,1375`) — все поля `readOnly`. Кнопки «Запланировать публикацию» и «Сохранить как черновик» не имеют handler-ов.

#### Существенные

5. **Числовые поля в моделях хранятся как строки** — `FinanceCalculation.cost`, `.markup`, `.total`, `.weight`, `.hours` и `MaterialReference.unit_price`, `.stock` — `CharField`, а не `DecimalField`/`IntegerField`. Это исключает автоматические вычисления себестоимости на стороне backend.

6. **Нет API для управления заказами** — `Order` создаётся через `OrderCreateView`, но отсутствуют endpoints для просмотра списка заказов, обновления статуса и просмотра деталей заказа.

7. **Нет аутентификации** — все API-endpoints открыты без авторизации. `CORS_ALLOW_ALL_ORIGINS = True` в `settings.py`.

8. **Поле `slug` продукта не включено в `ProductSerializer`** — фронтенд не может строить семантические URL для карточек товаров.

9. **Dashboard без метрик** — главная панель содержит только быстрый доступ к модулям, без KPI (число заказов, выручка, непрочитанные сообщения, ближайшие публикации).

10. **TypeScript-типы деградированы** — `BootstrapPayload` использует `typeof materialReference`, `typeof financeHistory` вместо именованных интерфейсов. При расширении данных типизация будет ломаться.

#### Незначительные

11. **Ссылки в боковой панели** (`Sidebar`) ведут на `#catalog`, а не на конкретные категории.
12. **Форма подписки в footer** — кнопка `type="button"` без handler-а.
13. **Экспортный путь в README** указывает `E:\Project\exports\google-sheets` — Windows-специфично.

### Этап 2 — Маркетинговый модуль: создание и редактирование (приоритет: высокий)

**Задачи:**

4. **Backend: CRUD для публикаций**
   - `POST /api/marketing/publications/` — создание
   - `PATCH /api/marketing/publications/<id>/` — редактирование
   - `DELETE /api/marketing/publications/<id>/` — удаление

5. **Frontend: редактируемый редактор постов**
   - Убрать `readOnly` с полей редактора
   - Подключить кнопку «Сохранить как черновик» → `PATCH /api/marketing/publications/<id>/`
   - Подключить «Запланировать публикацию» → `POST /api/marketing/publications/` + открыть диалог
   - После сохранения — обновить список постов через `loadBootstrap()`

### Этап 3 — Финансовый модуль: вычисления на стороне backend (приоритет: средний)

**Задачи:**

6. **Миграция типов полей**
   - `FinanceCalculation`: `cost`, `total` → `DecimalField`; `hours` → `DecimalField`; `markup` → `DecimalField` (процент)
   - `MaterialReference`: `unit_price` → `DecimalField`; `stock` → `DecimalField` + отдельное поле `unit` (кг, м², шт)
   - Создать новую миграцию с сохранением совместимости данных из Google Sheets

7. **Endpoint вычисления себестоимости**
   - `POST /api/finance/calculate/` — принимает `{product_id, material_name, quantity, hours}`, возвращает `{cost, total}` с применением актуального `unit_price` из `MaterialReference`

8. **UI-форма нового расчёта**
   - Заменить «макет Google Form» реальной формой с выбором материала из справочника
   - Автоматически заполнять поля «Стоимость материала» и «Стоимость работы» при выборе данных

### Этап 4 — Управление заказами (приоритет: средний)

**Задачи:**

9. **Backend: расширенный API заказов**

   - `GET /api/crm/orders/<id>/` — детальная карточка заказа
   - `PATCH /api/crm/orders/<id>/` — обновление статуса, срока, суммы, заметок

10. **Frontend: доска заказов (Kanban)**
    - Новый view-режим `orders` в навигации
    - Колонки по статусам: Новый → Подтверждён → В производстве → Готов → Доставлен
    - Карточка заказа с drag-and-drop или кнопками смены статуса
